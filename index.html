Introduction

Let's get started!

Contacts is a handy app to have. It stores all of your saved contacts. In this project, you will create one with your team. It helps you to understand and implement the basic programming concepts and principles of OOP.

Throughout the process, any questions/clarifications/recommendations to the Project Manager (your instructors) should be addressed by BAs. No one else from the team should be communicating with the PM unless absolutely needed.

Process

To get started, you need to do the following:

  1. DevOps will create a project on eclipse and push it to GitHub. Add your team members as collaborators on that repository.
  2. Team members will clone that repository on their eclipse (Check useful links tab for info on how to clone github repo onto eclipse)
  3. All team members will read and understand the requirements document fully
  4. Architect will discuss with the team members and come up with the design such as classes, attributes, methods, access modifiers for attributes, constructors , toString. Architect can write the design on a piece of paper or a shared document.
  5. Architect will discuss with team members and come up with the logic for each menu option
  6. Business Analyst will invite PM to join their breakout room and discuss the design solution.
  7. Once PM gives the go ahead, Team lead will assign work to each member
  8. Team members would need a baseline code to begin implementing features. Team lead / Architect can share their screen and implement the baseline code like creating the model/POJO classes, and a class that contains a phonebook list and the main method.
    Create these classes and push the code on “master/main” branch
  9. QA will pull this code, run the project. Initially the main method will be empty so the project won’t do anything. Once QA gives a go-ahead that the code so far is error-free and runs, move to the next step.
  10. Once step 7 is completed, each member of the team will create their own feature branch (give it a feature name like “search-functionality”) from the master/main branch. Create one branch per feature. Do not merge multiple features in the same branch. Don’t write your name in commits, feature names etc. I will be able to see who implemented which part of the code by viewing push and pull requests on the repository
  11. Each team member will implement their assigned work. This project is open book so feel free to use any resource online or otherwise. DO NOT copy paste code from anywhere.
  12. Once done implementing your assigned work, push your code to your branch, Open a pull request to merge your branch with master. Do not merge the pull request. Assign someone to review your code.
  13. Once your code is reviewed and you get a go ahead, merge your branch with master branch.
  14. Once all team members are done implementing their feature/s and merging with master, QA will pull the code , change branch to master and test by running your code several times. Any issues will be logged under issues tab on GitHub and QA/BA will assign the issue to a team member of their choice.
  15. Assigned team members will create a new branch from main, call it bug-fix-(issue-number), pull it on eclipse, switch to that branch and fix the issue. Once done, you can push the code and create a new pull request.
  16. Process continues until QA gives a go ahead to make your application live - i.e in our case, let the Project Manager know that the project is completed.

Requirements

In this project, you will write an application that creates and stores contacts. It can store information about people and organizations. In the app, you can store phone numbers of different people and different companies, like your favorite pizza shop or your bank.

There should be a base class Contact that contains information relevant to both an organization and a person, like name, phone number (which would be a String) and createdAt (which would be the datetime when the object was created example: 2022-03-22T22:39:53.737179) . Contact will have a stand-alone verifier that validates if a phone number entered by the user is valid or not. This method will potentially be used by different applications by passing the string phone number as a parameter. This method is stand-alone logic and hence can be called without having to create an object of Contact. A phone number is valid if it contains 12 characters.

Additionally, Person will have email and date of birth (date object) and Organization will have a website.

You can perform several actions in your phonebook application: [add, list, count, info, delete, search, exit]

Menu options

  1. add:
    • Asks the user whether they want to add a person or organization.
    • Based on their choice, display the questions pertaining to that Contact type.
    • Before building the object and storing it in the phonebook list, check if the phone number entered by the user is valid not not. Contact has a verifier that can check it for you. If the phone number is valid, retain the data entered by the user, if not, assign phone number to “[No Number]” text.
    • Create the contact object and store it in your phonebook list.
  2. list:
    • Display each contact in “contact id. Contact name” format
  3. count:
    • Display the number of records currently in the phonebook
  4. info:
    • Display the list of records
    • Ask the user to input contact id
    • If the contact id is present in the phonebook, display the contact information
  5. delete:
    • Display the list of records
    • Ask the user to input contact id
    • Delete the contact id from the phonebook
  6. search:
    • Slightly complex logic compared to others
    • Ask the user to enter a search query
    • Search through each contact in phonebook by checking if the contact name contains the search query
    • Display all contact names that contain the search result in id. contact name format
    • Ask instructor for HINT on which Collection Framework class to use to implement this functionality
  7. exit:
    • Exit the menu and end exection
